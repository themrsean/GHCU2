# GHCU2v2 Workflow (Project-wide)

## Project Overview
GHCU2v2 is a JavaFX desktop application for GitHub Classroom grading workflows. It loads assignment/comment definitions from JSON and provides UI tools to pull student repos, run checks (checkstyle/tests/imports), and generate grading reports.

## Entry Points
- `src/main/java/app/Launcher.java`: `main(String[] args)` entry point.
- `src/main/java/app/GitHubClassroomUtilities.java`: JavaFX `Application`; loads `/ui/MainWindow.fxml`.

## Core Package Responsibilities
- `app/`: Application startup and JavaFX launch.
- `ui/`: JavaFX controllers + FXML views (primary coordinator: `MainWindowController`).
- `service/`: Business logic and workflow orchestration.
    - `WorkflowEngine` + `WorkflowContext`: execute multi-step workflows.
    - `RunAllService`: coordinates full pipeline runs.
    - `service/steps/*`: step implementations (pull/extract/imports/reports/run-all).
    - Supporting services: `GitService`, `CheckstyleService`, `UnitTestService`, `ReportService`, etc.
- `model/`: POJOs representing assignments, rubrics, comments, mappings, reports.
- `persistence/`: Load/save settings and assignment data (JSON-backed stores).
- `util/`: Helpers (e.g., app data directory logic in `AppDataUtil`).

## Key Data Files and Resources
- JSON resources:
    - `src/main/resources/assignment.json`
    - `src/main/resources/assignments.json`
    - `src/main/resources/comments.json`
- JavaFX UI resources:
    - `src/main/resources/ui/*.fxml`
    - `src/main/resources/ui/grading.css`

## Dependency / Build Reality
- No Maven/Gradle project configuration in this repo.
- External dependencies are provided as local JARs under `lib/` (including JavaFX SDK jars and testing/checkstyle tooling).
- Compiled outputs may appear under `out/production/...` (treat as build artifacts, not source of truth).

## Architectural Data Flow (Mental Model)
1) Startup: `Launcher` → `GitHubClassroomUtilities` → loads `MainWindow.fxml`.
2) UI interactions are handled by controllers in `ui/` (especially `MainWindowController`).
3) Controllers delegate work to `service/` classes.
4) Workflow runs are executed by `WorkflowEngine` over `service/steps/*` using a `WorkflowContext`.
5) Persistence reads/writes settings/assignment data through `persistence/*Store` classes.
6) Reports are generated by `ReportService` (HTML/templating wrapper via `ReportHtmlWrapper`) and written into repo/work directories.

## Testing and Verification

### Test Location
- Tests live under `src/test/java` and should mirror production package structure.

### Framework
- JUnit 5 (JUnit Platform Console Standalone JAR exists under `lib/`).

### Policy
- Any feature addition or behavior change must include tests when feasible.
- Prefer tests for `model/`, `persistence/`, and `service/` logic over JavaFX controller/UI tests.
- After changes, tests must be compiled and executed. Failures must be resolved before considering work complete.

### Reporting
- Always state what was run and whether it passed.
- If tests cannot be added or executed, provide a concrete manual verification checklist.

### Testing Rules
- Any feature addition or behavioral change must include corresponding tests.
- Tests should target service/model layers, not JavaFX UI controllers, unless explicitly requested.
- Prefer deterministic, fast-running unit tests.
- Initial test target: `persistence.SettingsStore` default/load/save behavior.
- Prefer tests that use `Files.createTempDirectory(...)` and do not write into the repo.
- Default settings contract:
  - schemaVersion = 1
  - checkstyleConfigUrl = https://csse.msoe.us/csc1110/MSOE_checkStyle.xml

### Automated Test Execution
When test tools are available:
1. Compile tests.
2. Run all tests.
3. If failures occur:
  - Diagnose root cause.
  - Fix code (not tests) unless tests are incorrect.
4. Report:
  - What was run.
  - Number of tests passed/failed.
  - Relevant failure output.

### Test Design Expectations
- Tests should:
  - Assert behavior, not implementation details.
  - Cover edge cases (null input, empty collections, invalid paths).
  - Use realistic example assignment data where applicable.

### Test Enforcement Policy
- Any behavioral modification must:
  - "Compile production"
  - "Compile tests"
  - "Run tests"
  - "Resolve failures before concluding"
